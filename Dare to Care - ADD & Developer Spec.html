01 — Overview

What we are building

A focused web platform that replaces a stack of home-care paper forms with one digital, role-aware workflow — from PDF import to signed, stored record.

In one sentence Dare to Care is a forms platform for a home-care agency: Admins digitize and publish form templates from existing PDFs, Caregivers fill and sign those forms for assigned clients on mobile, and Office Managers review, track, and export the completed records — every submission keeping its template version, signatures, timestamps, and a branded PDF.

Design principlesWhat keeps this app focused

Data-driven, not hand-built

Every form is a JSON schema rendered by one shared field renderer. We never hand-code individual form screens — adding a form means adding a schema.

One source of truth for records

Caregiver submissions and Office Manager records read from the same store. No parallel demo arrays, no drift between roles.

Mobile-first for caregivers

The wizard is built for a phone in the field: one section at a time, large targets, safe autofill, explicit signature capture.

Audited & versioned by default

Every meaningful action writes an audit event. Every submission stores the exact template version it was completed against — historical PDFs are never overwritten.

NamingThis document is an ADD — App Design Document, not a GDD. This is a software product, not a game. All prior "GDD" language has been replaced.

App architecture

A conventional, buildable React + TypeScript structure. Business logic lives in services; pages stay thin; form templates live as data.

Organization rules

  • Keep business logic out of page components.
  • Form templates are JSON schemas, never one-off JSX.
  • One shared FieldRenderer handles every field type.
  • All permission checks read from permissions.ts.
  • Branded PDF generation lives in pdf.service.ts, not the wizard.
  • Keep source PDFs separate from completed-submission PDFs.
  • Dev seed data is isolated; production state starts empty.

Suggested stack

  • Vite + React 18 + TypeScript — replaces in-browser Babel.
  • React Router — role-gated routes.
  • Zustand or React Query — submission/template store.
  • pdf-lib (fill/stamp) or server-side render for PDFs.
  • Backend (to confirm): a real API + DB is required for any real client data — see Gaps & Risks.

04 — Screens & UX

Main screens & their jobs

Every screen has one clear job. Empty states teach the next step instead of showing fake demo filler.

Shared shell

All authenticated screens share one AppShell: a role-aware sidebar, a topbar (client/role context, search, account), toasts, and a footer status line (HIPAA-aware · Encrypted — UI affordance only; see Gaps). Caregiver screens collapse to a mobile layout.

Admin  Dashboard

Summarizes published templates, clients, submissions, and recent audit activity. When empty, shows getting-started steps (upload first PDF → publish → add clients).

Admin  Upload PDF

Drop/import a source PDF. System detects fields, tables, signatures, scoring, and autofill opportunities. Result is a draft — never auto-published.

Admin  Form Builder

AI interpretation summary, source preview, detected sections/fields, and a field inspector (label, type, required, options, scoring, autofill). Save draft / publish.

Admin  Users · Clients · Audit

User & permission management, the shared client directory, and a filterable audit log of every recorded action.

Caregiver  My Day / Available Forms

Today's assigned clients and due forms; a browseable list of published templates filtered to what the caregiver can complete.

Caregiver  Form Wizard

Pick client → step through sections → accept/reject autofill → required validation → computed score/tier → review → branded PDF preview → sign → submit.

Office Mgr  Records / Submissions

Search & filter by client, caregiver, form, status, and date. Open submission detail / PDF, request correction, export when permitted.

Office Mgr  Team & Supervisory

Team visibility and supervisory visit documentation; monitors recurring required forms (care plan review, supervisory visit, activity report, fall-risk reassessment).

05 — User Flows

The three core flows

Import & publish, complete & submit, review & export — the loop that makes the platform work.

Flow A · AdminPDF → published template

Open Upload PDF & import a source form

Admin uploads or selects a reference PDF from the library.

System extracts structure

Title, sections, field labels & types, required flags, scoring rules, table columns, signature lines, and layout hints become a draft schema.

Admin reviews the interpretation

Purpose, audience, record type, cadence, output, and suggested autofill fields — each shown with source & confidence.

Admin edits fields in the builder

Adjust labels, types, options, required flags, scoring, and table columns via the field inspector.

Save draft or publish

Publishing versions the template and makes it available to caregivers and visible in Office Manager records. An audit event is written.

Flow B · CaregiverFill & submit a form

Choose a form & client

From My Day or Available Forms, then select the assigned client.

Step through sections

One section per step. Safe autofill (client profile, today's date, existing records) can be accepted individually or in bulk.

Required fields gate progress

The wizard blocks advancing until required fields are valid. Computed fields (e.g. fall-risk score & tier) update live.

Review, sign, preview

Review answers, capture signature explicitly (never auto-signed), and preview the branded PDF.

Submit & file

Submission stores template version, client, values, signer, timestamps, and a generated PDF reference. Audit event written.

Flow C · Office ManagerRecords review

Open Submissions

Reads from the same shared submission store caregivers write to.

Search & filter

By client, caregiver, form type, status, and date range.

Open detail / PDF

Check completion, request correction if needed, export/download if authorized.

Monitor recurring forms

Track required cadences: care plan review, supervisory visit, activity report, fall-risk reassessment.

Drafts & historyAn interrupted wizard auto-saves a draft tied to (caregiver, template version, client). Submissions are immutable once filed; corrections create a new linked submission rather than editing history.

Validation & scoringRules the renderer enforces

Required gating

A section cannot be left until its required fields are valid. validation.service.ts derives all rules from the schema — never hand-written per form.

Type validation

Dates parse to ISO; select/radio values must be a defined option; table rows validate per-column; signature must be non-empty ink.

Computed fields

formula:"sum" totals option scores; formula:"tier" maps the total to a risk band. Computed fields are read-only and recalc live.

Autofill safety

Safe sources (client profile, today's date, prior records) can auto-apply. Signatures never auto-fill — they may suggest a name but require explicit ink. Source + confidence shown.

PacketsSplitting the big forms

Client Admission Packet (25pp → modules)

  1. Welcome & agreement acknowledgement
  2. Client profile & contact info
  3. Service agreement & schedule
  4. Rate / payment authorization
  5. Client assessment
  6. Client care plan
  7. Rights / HIPAA / confidentiality acks
  8. Advance directives & emergency prep
  9. Vehicle authorization
  10. Final signature review

New Hire Paperwork (22pp → modules)

  1. Interview questions
  2. Job descriptions & acknowledgements
  3. Availability & emergency contacts
  4. Rules of the road acknowledgement
  5. Handbook acknowledgement
  6. Personal-care competency / skills
  7. Automobile use policy
  8. Time-off request
  9. HIPAA / confidentiality policy
  10. Emergency preparedness · final signatures

Packets aren't one giant screenEach module is its own completable form with its own PDF (or one combined packet PDF once all modules are done). A packet tracks per-module completion so a caregiver/client can resume.

StorageHow completed forms are stored

  • A completed form becomes a Submission record (see Data Model) capturing templateId + templateVersion, client, caregiver, all values, computed values, signatures, status, timestamps, and a pdfUrl.
  • Submissions are immutable. A correction creates a new linked submission; the original and its PDF are preserved.
  • Source/reference PDFs (admin imports) are stored separately from generated submission PDFs.
08 — Data Model & Types

Data model & TypeScript types

Six core entities. Define them once in src/types/ and import everywhere — no ad-hoc shapes.

User

Identity, role, permissions, session.

Client

Care recipient + profile used for autofill.

FormTemplate

Versioned schema of sections & fields.

FormField

One field: type, options, autofill, validation.

Submission

A completed form + PDF + signatures.

AuditEvent

Immutable record of who did what, when.

src/types/index.ts 
export type Role = "admin" | "officeManager" | "caregiver";
export type FieldType =
  | "text" | "textarea" | "date" | "radio" | "select"
  | "checkbox" | "table" | "computed" | "signature" | "policyText";

export interface User {
  id: string; name: string; email: string;
  role: Role; status: "active" | "invited" | "disabled";
  permissions: string[]; createdAt: string; lastLoginAt?: string;
}

export interface Client {
  id: string; name: string; dob: string; mrn?: string;
  phone?: string; address?: string; primaryPhysician?: string;
  emergencyContacts: Contact[]; allergies?: string;
  carePlanSummary?: string; assignedCaregivers: string[];
  status: "active" | "inactive"; createdAt: string; updatedAt: string;
}

export interface FieldOption { label: string; score?: number; }
export interface Autofill {
  source: string; from: string; confidence: number; safe: boolean;
}
export interface FormField {
  id: string; label: string; type: FieldType; required?: boolean;
  semantic?: string;                 // maps value to a known concept
  options?: FieldOption[];          // radio / select
  columns?: TableColumn[];          // table
  formula?: "sum" | "tier";       // computed
  autofill?: Autofill;
  validation?: ValidationRule;
  layoutHints?: Record<string, unknown>;
}

export interface FormSection { id: string; title: string; order?: number; fields: FormField[]; }

export interface FormTemplate {
  id: string; key: string; name: string; category: string;
  description?: string; sourcePdfId?: string;
  version: number; status: "draft" | "published" | "archived";
  interpretation?: Interpretation; sections: FormSection[];
  createdBy: string; publishedAt?: string; updatedAt: string;
}

export interface Submission {
  id: string; templateId: string; templateVersion: number;
  clientId: string; caregiverId: string;
  values: Record<string, unknown>;          // fieldId → value
  computedValues: Record<string, unknown>;  // score, tier…
  signatures: Signature[];
  status: "draft" | "submitted" | "needsCorrection" | "reviewed";
  pdfUrl?: string; submittedAt?: string; reviewedAt?: string;
  correctionOf?: string;             // links a correction to its original
}

export interface AuditEvent {
  id: string; timestamp: string; actorId: string; actorRole: Role;
  action: string; targetType: string; targetId: string;
  ip?: string; metadata?: Record<string, unknown>;
}

Audit eventsWhat gets logged

Write an AuditEvent for: sign in/out · client created/edited · PDF uploaded/imported · template draft saved · template published/unpublished · form submitted · submission reviewed/exported/downloaded · user/permission changes.

09 — PDF & Export

PDF generation & export

Today the prototype only previews a PDF. The real pipeline must generate, brand, and persist a file per submission.

Rules

  • Every submitted form generates or stamps a branded PDF.
  • The PDF stores the exact template version used.
  • Never overwrite historical PDFs when a template changes.
  • For packets: one PDF per completed module, or one combined packet PDF after all modules complete.
  • Keep generated submission PDFs separate from imported source PDFs.

Implementation options

  • Client-side (pdf-lib / @react-pdf/renderer) — fast MVP, fine for simple 1-page forms.
  • Server-side (headless render) — better for fidelity, large packets, and keeping PHI off the device. Recommended once a backend exists.
  • Either way, the renderer reads the schema — branded layout is shared, not per-form code.

Decision neededWhere are generated PDFs hosted, and for how long are they retained? This depends on the backend/compliance choice — see Gaps & Risks.

11 — Gaps & Risks

Gaps, risks & what to clarify

What's missing, confusing, or risky in the current prototype — and the open questions I need answered before development locks in.

Prototype gaps to clean up

Records not wired to live data

Office Manager records use older static APP_DATA paths in places. Must read the shared submission store.

PDF is preview-only

No persisted generate→store pipeline yet. Needs pdf.service.ts + storage.

Only 2 forms modeled

Fall Risk & Medication List exist as schemas; the rest need schemas built from the reference PDFs.

Browser-only prototype

JSX-through-Babel is fine for demo; graduate to a real project to go further.

Role switching ≠ auth

The login is a prototype shortcut. Real authentication & session required.

Compliance is UI-only

"HIPAA-aware · Encrypted" is a label, not implemented compliance. Do not treat as done.

Security & privacy — treat as blockingThis app handles PHI (names, DOB, MRN, medications, clinical assessments). Before any real client data: real authentication & RBAC enforced server-side, encryption in transit & at rest, access-controlled PDF storage with retention rules, complete audit logging, and a HIPAA review (likely a BAA with any vendor/host). The current prototype meets none of these and must not hold real PHI.

Confusing / duplicated parts

  • Two data paths for records (static demo vs. live store) — collapse to one.
  • Form structure partly lives in UI code and partly in data.js — move all form definitions to data/formSchemas/.
  • "Owner/role" on forms mixes who the form is about (Client) with who fills it (Caregiver/OM) — split into subjectType vs completedByRole.
